# Founder Console Integration Contract ÔÇö Minimal Transport Design

**Type:** Design Specification
**Status:** Approved
**Date:** 2026-07-19
**Approved:** 2026-07-19 (Founder)
**Depends on:** ADR-002 (Accepted); ADR-014 (Accepted); `FounderConsolePort`, `FounderApprovalRequest`, `FounderApprovalDecision`, `CandidateDeployment` (all unchanged, all Approved-equivalent)

This document defines *behavior*, not a framework, library, or wire format. It describes what a compliant concrete adapter must send, receive, verify, and refuse ÔÇö not how (no HTTP library, no specific serialization, no implementation code is specified or implied). Where a field is not currently obtainable from the existing, approved contracts, this document says so explicitly rather than inventing a source for it.

---

## 1. Request Semantics

The Deployment Agent invokes `FounderConsolePort.submitCandidateForApproval(request: FounderApprovalRequest)`, where `FounderApprovalRequest = { candidate: CandidateDeployment }` ÔÇö unchanged, per ADR-014 ┬º7. Everything the wire-level request to Founder Console carries beyond that single field must come from one of two places: **already present** in reachable, existing data, or **synthesized by the adapter itself**, entirely internal to OP-02's implementation, never requiring a change to `FounderConsolePort` or `FounderApprovalRequest`.

| Field this document requires | Status | Source |
|---|---|---|
| Candidate deployment identifier | **Already available** | `candidate.id` (from `GitHubIntegrationPort`, OP-01 ÔÇö a real commit SHA) |
| Repository/project identity | **Already available** | `candidate.artifactRef` (OP-01 constructs this as `<repositoryUrl>#<branch>@<sha>` ÔÇö already carries repository and branch identity as a composite string; not a separately structured field) |
| Target environment | **Not currently available** | `DeploymentLifecycleRequest.characteristics` is deliberately typed `unknown` ÔÇö the Deployment Agent has no visibility into environment, change type, or any of ADR-007's fourteen classification signals; that information is Policy-Engine-internal by design. Transmitting it to Founder Console would require a `FounderApprovalRequest` (or `DeploymentLifecycleRequest`) contract change. **Flagged as a proposed future addition, not defined by this document.** |
| Risk classification | **Implicit, not transmitted as a field** | `submitCandidateForApproval()` is only ever invoked on the high-risk path (`executeHighRiskDeployment` "operates on a candidate already classified high-risk" ÔÇö orchestrator.ts). Every request this adapter ever sends is therefore high-risk by construction. No field is needed to convey this; a compliant adapter does not need ÔÇö and this document does not require ÔÇö a `riskClassification` field on the wire request. |
| Immutable request identifier | **Adapter-synthesized** | Generated once per call, entirely inside the concrete adapter (mirroring how OP-01's adapter constructs `artifactRef` internally without any contract change). Never exposed through `FounderConsolePort`'s own type signature. |
| Request creation timestamp | **Adapter-synthesized** | Set once, at submission, by the adapter. |
| Expiry timestamp | **Adapter-synthesized** | Set once, at submission, by the adapter, as creation timestamp plus a bounded validity window (see ┬º3.7 of ADR-014; the exact window is an implementation parameter, not fixed here). |
| Approval nonce | **Deployment-Agent-synthesized** | A single-use, cryptographically unpredictable value **generated by the Deployment Agent when the approval request is created**, included in the approval transport request, and required to be bound unchanged into the Founder Console decision response. The Deployment Agent verifies the returned nonce and marks it consumed only after successful verification; a response whose nonce is missing, altered, mismatched, or previously consumed is rejected as an adapter failure (ADR-014 ┬º3.9). Generated only by the Deployment Agent ÔÇö Founder Console echoes it back, never generates it. Not a domain-decision field ÔÇö exists purely as transport-level replay protection. |
| Decision timestamp | **Founder-Console-synthesized** | Set by Founder Console when it captures the Founder's decision, returned as part of the response, checked by the adapter against the allowed validity window (ADR-014 ┬º3.9, check 6). Distinct from the approval nonce: this is the one piece of transport metadata Founder Console itself generates; the adapter does not generate it, only validates it. |

**All of the above ÔÇö request identifier, approval nonce, request creation timestamp, expiry timestamp, and decision timestamp ÔÇö are transport-level metadata only.** None of them are additions to `FounderApprovalDecision`, `FounderApprovalRequest`, `CandidateDeployment`, or any other existing domain contract, and none of them are ever exposed through those types' own signatures. Their sole purpose is stronger replay protection, audit traceability, and forensic evidence for this one transport exchange (ADR-014 ┬º3.9ÔÇô┬º3.10) ÔÇö not a change to what the Deployment Agent's orchestration logic receives or reasons about, which remains exactly `FounderApprovalDecision = "approved" | "rejected"`.

**Requirement:** a compliant adapter must not invent or transmit any field beyond `candidate.id`, `candidate.artifactRef`, the Deployment-Agent-synthesized request identifier, approval nonce, and creation/expiry timestamps, and the Founder-Console-synthesized decision timestamp, unless a future ADR explicitly extends `FounderApprovalRequest` or `DeploymentLifecycleRequest` to carry more (e.g. target environment). This document does not authorize that extension.

## 2. Decision Semantics

The domain decision returned to the Deployment Agent's caller remains exactly `FounderApprovalDecision = "approved" | "rejected"` ÔÇö unchanged, per ADR-014 ┬º7. No third domain value is introduced by this document, and none may be introduced by any implementation of it.

**Operational outcomes are not domain decisions.** The following are adapter failures, never converted to `"approved"` or `"rejected"`:

- Timeout (no response within the bounded window).
- Founder Console unavailable (connection refused, DNS failure, 5xx-equivalent, or any transport-level failure).
- Malformed response (fails to parse, fails schema validation, or is otherwise not interpretable as a decision).
- A response whose integrity cannot be verified (┬º3, Security Semantics).
- A response bound to a different request identifier or a different candidate than the one outstanding (┬º3.10 of ADR-014).
- A response received after its own expiry timestamp has passed.
- Any response that is well-formed but does not resolve to exactly one of `"approved"` or `"rejected"` (e.g., a genuinely third value, or an ambiguous/partial response) ÔÇö treated as malformed, not as a new domain state.

Every one of the above must surface to the Deployment Agent's caller as a failure distinguishable from a genuine domain decision (per ADR-014 ┬º3.6), and must prevent deployment ÔÇö the high-risk orchestration path must not proceed to `DeploymentExecutionPort` on any of them, exactly as it already must not proceed on an explicit `"rejected"`.

## 3. Security Semantics

- **Founder Console is the sole identity-verification and allowlist authority.** The adapter performs no identity check, no Cloudflare Access handshake, and no allowlist logic of its own ÔÇö it trusts Founder Console's response only to the extent ┬º3's remaining requirements (integrity, binding, expiry) are satisfied, and never substitutes its own judgment for identity verification it is not positioned to perform.
- **Authenticated and integrity-protected communication.** Whatever transport is ultimately selected (┬º4) must authenticate the Deployment Agent to Founder Console (so an arbitrary caller cannot submit requests or read decisions) and must let the adapter verify that a decision response actually originated from Founder Console and was not altered in transit. This document does not select a specific mechanism (mutual TLS, signed responses, a shared secret, or another approach) ÔÇö that is implementation detail for OP-02, constrained only by this requirement.
- **Decision binding.** Per ADR-014 ┬º3.9ÔÇô┬º3.10, a valid approval response must satisfy all six of the following, conjunctively: (1) request identifier matches; (2) candidate identifier (`candidate.id`) matches; (3) candidate artifact reference (`candidate.artifactRef`) matches; (4) the request has not expired; (5) the approval nonce returned matches exactly what the Deployment Agent generated for this request, and has not previously been marked consumed; (6) the decision timestamp falls within the allowed validity window relative to the request's creation timestamp. Failure of **any single one** of these six checks is an adapter failure ÔÇö it must never become `"approved"` and must never become `"rejected"`.
- **Expiry.** Check (4) above: a decision received after its own expiry is an adapter failure, never `"rejected"`.
- **Replay prevention.** Checks (1) and (5) above: each request identifier and each approval nonce are single-use; a decision bearing an identifier the adapter is not currently awaiting, or an approval nonce that is missing, altered, mismatched, or already marked consumed, is refused outright ÔÇö never accepted a second time for a new orchestration call. The Deployment Agent generates the approval nonce and is the sole party that marks it consumed, immediately upon successful verification.
- **Fail-closed behavior.** Per ADR-014 ┬º3.6: every failure mode above ÔÇö including failing any one of the six binding checks ÔÇö prevents deployment; none defaults to a domain decision.
- **Safe error reporting.** Every error this adapter produces identifies the failed operation and the candidate/request identifiers involved, but never the raw transport-level authentication material (however ┬º4's recommendation ultimately authenticates the call), mirroring OP-01's `GitHubIntegrationAdapterError`/`sanitizeRepositoryIdentifier` precedent.
- **No credentials or tokens in audit events.** The `AuditTrailEntry.detail` this adapter's audit-recording composes (per ADR-014 ┬º3.8) must carry only the fields already covered by this document's Request/Decision Semantics ÔÇö never transport-level secrets.
- **No direct LLM, MCP, or AI-surface access.** Per ADR-014 ┬º3.2ÔÇô3.3: nothing about this transport contract may be exposed as a callable MCP tool, a chat-invokable action, or any other AI-agent-reachable surface. It is reachable only from the deterministic Deployment Agent code path that already calls `FounderConsolePort` (WP-09's `executeHighRiskDeployment`).

## 4. Interaction Model

`FounderConsolePort.submitCandidateForApproval()` is synchronous ÔÇö it returns `FounderApprovalDecision`, not `Promise<FounderApprovalDecision>` ÔÇö and per this task's constraints, that interface is not to be changed. A human Founder's response time is not itself bounded by anything in the approved architecture, which creates real tension with a synchronous, must-eventually-return method. Four options were compared.

### Option 1 ÔÇö Bounded synchronous request/response (single call, server-held)

One logical call: the adapter submits the request and makes a single blocking call to a "wait for decision" operation that the Founder Console itself holds open (server-side) until a decision exists or its own bound is reached, mirroring how OP-01's `execFileSync` blocks on a single `git` invocation.

- **Compatibility with current port:** Full ÔÇö one call in, one value out, no port change.
- **Operational risk:** The single call ties up the calling process/thread for the entire wait; if the bound is set generously to tolerate normal human response time (minutes), that is a long single blocking hold.
- **Timeout implications:** One timeout value governs the entire wait; straightforward to reason about, but a single network hiccup near the end of a long wait discards the whole wait.
- **Deployment complexity:** Low ÔÇö no additional process, but requires Founder Console to support a server-held/long-held connection, which is Founder Console's own implementation concern, not this repository's.
- **Architecture impact:** None.
- **Auditability:** One request event, one outcome event ÔÇö coarse-grained (no visibility into the wait itself).
- **Security implications:** Fewer round trips than Option 2 means a smaller window of individual exchanges to secure, but a single long-held authenticated connection has its own operational exposure (a connection held open for minutes is a larger target than several short ones).

### Option 2 ÔÇö Bounded, client-driven long-polling

The adapter submits the request once, then repeatedly makes short, independent status checks (each its own short-timeout call) at an interval, until a decision appears or the overall bound is reached ÔÇö the same "bounded external call, looped, with an overall ceiling" shape OP-01 already established for `git`, extended to a loop.

- **Compatibility with current port:** Full ÔÇö the entire loop happens inside one synchronous method call; the port sees a single call in, one value out.
- **Operational risk:** Lower per-attempt risk than Option 1 (any single failed poll doesn't waste the whole wait ÔÇö the loop simply continues, up to the overall bound); most consistent with OP-01's already-reviewed and CTO-approved pattern.
- **Timeout implications:** Two parameters instead of one (per-poll timeout, overall ceiling) ÔÇö marginally more to configure and validate, but each is individually easy to reason about and bound, following the same finite-positive-integer-with-a-maximum discipline OP-01 established.
- **Deployment complexity:** Low ÔÇö no additional process; Founder Console only needs a simple status-check endpoint, not a long-held connection.
- **Architecture impact:** None.
- **Auditability:** Finer-grained than Option 1 if desired (each poll can optionally be observed), though this document does not require recording every individual poll ÔÇö only the overall submission and outcome, per ADR-014 ┬º3.8.
- **Security implications:** More individual authenticated exchanges than Option 1, each short-lived ÔÇö a smaller individual exposure window per exchange, more exchanges overall.

### Option 3 ÔÇö Local deterministic broker/helper process

A separate, already-running local process maintains the actual connection to Founder Console (by whatever means, potentially long-held or event-driven) independently of any single deployment attempt. The synchronous `FounderConsolePort` adapter's only job is a fast, local, bounded call to that broker: "do you have a decision for request X yet?"

- **Compatibility with current port:** Full ÔÇö the port-facing call is a fast local query, trivially boundable.
- **Operational risk:** Decouples "wait for a slow human" from "block a Node process" entirely ÔÇö the broker can track many outstanding requests independently of any single deployment invocation. Introduces a new operational dependency: the broker itself must be running, monitored, and restarted on failure.
- **Timeout implications:** The port-facing timeout can be very short and reliable (a local call); the broker's own wait against Founder Console is a separate concern with its own, likely more generous, bound.
- **Deployment complexity:** Highest of the four ÔÇö a new persistent process to build, deploy, and operate, with its own lifecycle independent of any single deployment run.
- **Architecture impact:** Arguably introduces a new operational component, even if not a new architectural one in the Architecture Specification's five-component sense ÔÇö worth flagging for CTO judgment on whether this itself needs its own review.
- **Auditability:** Best of the four ÔÇö the broker can maintain a durable, independent record of every outstanding request regardless of whether any individual deployment invocation ever comes back to collect it.
- **Security implications:** Two boundaries to secure instead of one (adapter-to-broker, broker-to-Founder-Console) ÔÇö more surface area, but each individual boundary can be narrower (adapter-to-broker can be a trusted local-only channel).

### Option 4 ÔÇö Change `FounderConsolePort` to asynchronous

`submitCandidateForApproval(): Promise<FounderApprovalDecision>`. The technically cleanest fit for a fundamentally async problem (waiting on a human).

- **Compatibility with current port:** **None ÔÇö this is the interface change this task's constraints explicitly forbid implementing.** Ruled out for OP-02 by construction, listed here only because the CTO's own instructions required this comparison.
- **Operational risk:** Lowest of the four for the underlying problem itself ÔÇö no blocked thread/process, natural fit for an indefinite-latency wait.
- **Timeout implications:** Cleanest to express (a `Promise` with its own cancellation/timeout semantics).
- **Deployment complexity:** Low, once done ÔÇö but "once done" understates the cost below.
- **Architecture impact:** **Large.** `DeploymentAgent`'s constructor, `executeHighRiskDeployment`, every `...WithAudit` composed method that touches it, `runDeploymentLifecycle`'s entire synchronous composition (ADR-013), and WP-07/WP-09's existing CTO-approved contracts would all need rework. This is not a boundary-local change ÔÇö it ripples through the already-approved orchestration core.
- **Auditability:** Unaffected in principle, but the surrounding rework is large enough that a regression is more likely than under Options 1ÔÇô3.
- **Security implications:** No inherent security difference from Options 1ÔÇô2 at the transport level ÔÇö the risk here is entirely architectural blast radius, not security.

### Comparison Summary

| | Port change | New process | Timeout model | Architecture impact |
|---|---|---|---|---|
| 1. Bounded sync (server-held) | No | No | Single value | None |
| 2. Bounded long-polling | No | No | Two values | None |
| 3. Local broker | No | **Yes** | Two-tier | Operational, not architectural |
| 4. Async port | **Yes** | No | Cleanest | **Large ÔÇö separate approval required** |

### Recommendation (not an approval)

**Option 2 (bounded, client-driven long-polling)** is recommended: it requires no port change, no new persistent process to operate and secure, and directly reuses the timeout-and-validation discipline OP-01 already established and had CTO-reviewed. It is more resilient than Option 1 (a single failed exchange doesn't discard the whole wait) without Option 3's added operational component, and it avoids Option 4's out-of-scope architecture change entirely.

**Bounded, client-driven long-polling (Option 2) is the Approved interaction model**, adopted as part of this document's Founder approval on 2026-07-19. Concrete wire-level detail ÔÇö endpoints, payload shapes, authentication, and integrity signing ÔÇö is delegated to the companion `Founder-Console-Wire-Protocol-Specification.md` (Approved), not defined here. If, in practice, Founder response latency turns out to make Option 2's bounded wait routinely insufficient, Option 3 is the next-best candidate specifically because it does not require reopening `FounderConsolePort` ÔÇö that trade-off is noted here for a future decision, not resolved by this document.

## 5. Future-Proofing Note (Non-Normative)

*The following is not a requirement of this document, not part of the MVP, and not required for Operationalization. It is recorded only as a possible future evolution.*

A future production-hardening pass may introduce a **signed, immutable Approval Token** ÔÇö a durable, independently verifiable artifact of the Founder's decision, for external audit and compliance purposes beyond this project's own audit trail. This document does not define such a token's format, signing mechanism, storage, or verification procedure. The transport-level metadata already required by ┬º1 and ┬º3 (request identifier, candidate identifier, artifact reference, request creation timestamp, approval nonce, decision timestamp) would give such a token a natural set of fields to bind to if it is ever built ÔÇö noted here purely as an observation about future extensibility, not a specification, and not something OP-02 is required or expected to implement.

## 6. Non-Goals

This document does not:

- Select a serialization format, HTTP library, or specific transport technology.
- Define the concrete adapter's TypeScript shape, class structure, or file layout ÔÇö that is OP-02 implementation, once ADR-014 and this document reach Accepted/Approved status.
- Extend `FounderApprovalRequest` or `DeploymentLifecycleRequest` to carry target environment or classification rationale ÔÇö flagged in ┬º1 as a possible future addition, not defined or authorized here.
- Fix specific timeout values, polling intervals, or expiry windows ÔÇö implementation parameters for OP-02, bounded by the discipline this document and ADR-014 require, not fixed numbers.
- Approve Option 2 or any other interaction model.
- Define or require the signed Approval Token mentioned in ┬º5 ÔÇö explicitly future, non-normative, and out of scope for MVP/Operationalization.
