## 1. Product Identity

Independent AI platform for financial understanding. Not a bank, not a financial advisor, not a sales platform. Helps users understand financial decisions before deciding. Final decision always belongs to the user. Product Bible has no substantive content beyond metadata (v0.1, Draft) — not a usable source for identity claims.

---

## 2. Product Goals

Reduce uncertainty. Increase clarity. Build trust through understanding. Never persuade.

---

## 3. Brand Rules

Personality: calm, intelligent, patient, transparent, respectful, approachable.
Never: arrogant, sales-oriented, manipulative, overwhelming, intimidating.
Emotional target: "I finally understand this" — not "I hope this is correct."
Trust is earned through behaviour (honesty, transparency, consistency, clarity), never through visual effects.

---

## 4. Design Rules

Governing principle: Understanding Builds Trust.
Simple means obvious, not empty. If a design needs explaining, it has failed.
Calm over urgent, aggressive, or overwhelming.
Minimize cognitive load; users spend attention on financial understanding, not on learning the interface.
Progressive disclosure over showing everything at once.
Remove before adding — every element must justify its existence.

---

## 5. Visual Rules

Feel: trustworthy, calm, modern, professional, approachable. Never corporate or intimidating.
Hierarchy via typography, spacing, contrast, grouping — not colour.
Icons never replace text; if an icon needs explanation, drop it.
Illustrations and charts educate, never decorate. Every chart must answer a specific question.

---

## 6. Typography Rules

Primary communication tool; readability over expressiveness.
Small, consistent type scale.
Break dense text into short paragraphs, lists, examples.
Financial numbers (amounts, rates, payments, deadlines) get visual emphasis.
Same heading style always signals the same importance level.

---

## 7. Color Rules

Function over decoration. Neutral foundation, limited purposeful accents.
Consistent meaning-to-colour mapping across the product.
Never the sole communicator of state or meaning — pair with text, icon, or layout.

---

## 8. Layout Rules

Whitespace is functional, not filler.
Defined grid, rarely broken.
Alignment and grouping signal relationships between elements.
Prefer more screens over crowded ones.
One clear focal point per screen.
Responsive by default — understanding never depends on screen size.

---

## 9. Component Rules

One problem per component. Reusable across all modules (Housing Loans, Leasing, Document Analysis, Financial Terms, future modules). Identical behaviour everywhere it appears. Minimal exposed complexity; reveal advanced functionality progressively. Usable via keyboard, touch, and mouse.

---

## 10. AI Rules

AI is the interface, not a feature bolted onto it. Always available, never intrusive. Primary guide; navigation is secondary.
Explains its reasoning, states what information it used, states uncertainty.
Guides, never controls or pressures decisions.
On uncertainty: state it clearly, suggest a next step — never guess.

---

## 11. Accessibility Rules

Not optional. Design for varying age, experience, technical knowledge, ability, financial confidence.
Never colour-only signaling.
Interactive elements: identifiable, activatable, forgiving of mistakes.
Minimize thinking, remembering, and decision load per screen.
Prevent errors before they occur.

---

## 12. Non-Negotiable Rules

- User makes the final decision, always.
- Colour is never the sole meaning-carrier.
- Decoration never competes with understanding.
- Beauty never reduces usability.
- AI never interrupts, pressures, manipulates, or distracts.
- No urgency, pressure, or manipulation anywhere in the interface.
- No UI element added purely because it's fashionable.
- Design authority order is fixed: Constitution → Brand DNA → Design Bible → Product Bible → Product Specifications.

---

## 13. Design Decision Checklist

- Improves understanding?
- Reduces cognitive load?
- Strengthens trust?
- Matches Brand Rules?
- Consistent with Design Rules?
- Any meaning colour-only? (must be no)
- Keyboard / touch / mouse usable?
- One clear focal point, hierarchy via structure not decoration?
- Simpler alternative available? (use it)
- Every element justified?

## 14. Approved Design Constraints

- Design authority order is fixed (see Non-Negotiable Rules) and may not be reordered without a Founder-approved amendment.
- No Design Tokens system is defined in any source document, despite Design Foundation Specification requiring components to "follow Design Tokens." Do not invent one — flag if a design decision is blocked by this gap.
